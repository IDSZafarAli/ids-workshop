# Attachment Naming & Retrieval

RavenDB attachments are identified by `(documentId, name)`. The naming scheme has to be stable, predictable per variant, and unambiguous across feature types.

## Naming Convention

```
<kind>-<id>-<variant>
```

| Example                              | Meaning                                       |
| ------------------------------------ | --------------------------------------------- |
| `photo-3f8a1c92-original`            | Photo variant `original`, photoId `3f8a1c92`  |
| `photo-3f8a1c92-thumbnail`           | Same photo, thumbnail variant                 |
| `profile-original`                   | User profile photo (one only — no id segment) |
| `wo-doc-9d4f2-original`              | Work-order document, attachment id `9d4f2`    |

Conventions:

- `kind` is the feature noun (`photo`, `profile`, `wo-doc`).
- `id` is a `randomUUID()` generated by the service when storing.
- `variant` is the variant name from the `ImageVariant.name` field.
- Single-instance attachments (one profile photo per user) skip the `id` segment.

The naming is **owned by the domain service**. `AttachmentService` doesn't know or care.

## Storing

```typescript
import {randomUUID} from 'node:crypto';

const photoId = randomUUID();
for (const variant of result.variants) {
  this._attachmentService.store(
    session,
    part.id,
    `photo-${photoId}-${variant.name}`,
    variant.buffer,
    'image/webp',
  );
}
part.photos.push({photoId, isDefault, rotation, description, tags, ...});

await session.saveChanges();   // document + attachments commit together
```

The `photoId` is also the entity-side reference — the `Part.photos[]` array stores `{photoId, ...metadata}` and the URL includes it for retrieval.

## Retrieval — Stream + Dispose

`AttachmentService.getStream` returns a Node `Readable` plus a `dispose` callback. The session and result lifecycle is wrapped — the caller streams, then disposes.

```typescript
@Get(':partNumber/photos/:photoId')
public async getPhoto(
  @Param('partNumber') partNumber: string,
  @Param('photoId') photoId: string,
  @Query('variant') variant: string = 'original',
  @Res({passthrough: true}) res: Response,
): Promise<StreamableFile> {
  const result = await this._partService.getPhotoStream(partNumber, photoId, variant);
  if (!result) {
    throw new NotFoundException('Photo not found');
  }

  res.set({
    'Content-Type': result.contentType,
    'Cache-Control': 'private, max-age=3600',
  });
  return new StreamableFile(result.stream, {
    onDestroy: result.dispose,    // dispose when the stream closes
  });
}
```

Service side:

```typescript
public async getPhotoStream(
  partNumber: string,
  photoId: string,
  variant: string,
): Promise<AttachmentStreamResult | null> {
  const docId = `parts/${partNumber}`;
  const name = `photo-${photoId}-${variant}`;
  return this._attachmentService.getStream(docId, name);
}
```

The `StreamableFile.onDestroy` hook fires after the response is sent or the request aborts — the RavenDB session and the attachment result both get disposed.

## Read Endpoint Caching

```typescript
res.set({
  'Cache-Control': 'private, max-age=3600',
  'ETag': `"${photoId}-${variant}"`,
});
```

`private` because attachments are tenant-scoped — never cache in shared/public caches. `ETag` lets browsers conditional-GET on the next request; respond with 304 when `If-None-Match` matches.

## Buffered Read (Small Files Only)

For sub-100KB attachments where streaming complexity isn't worth it:

```typescript
const result = await this._attachmentService.get(docId, name);
if (!result) throw new NotFoundException();
return {
  data: result.data,
  contentType: result.contentType,
};
```

This buffers the full attachment into memory. Use only for small icons or thumbnails.

## Deleting

```typescript
this._attachmentService.delete(session, docId, attachmentName);
// ... more deletes ...
await session.saveChanges();
```

When a parent document is soft-deleted, **don't** automatically delete attachments — they preserve the audit trail. Hard-delete attachments only when the parent is hard-deleted (rare in this project) or when the user explicitly removes the photo.

## Removing Photos in an Update Flow

```typescript
const submittedIds = new Set(dto.photos?.map((p) => p.photoId) ?? []);

for (const existing of part.photos ?? []) {
  if (!submittedIds.has(existing.photoId) && !skipIds.has(existing.photoId)) {
    // Photo was on the server, not in the submission, not in skipIds — remove it
    for (const variant of PartService._PHOTO_VARIANTS) {
      await this._attachmentService.delete(
        session,
        part.id,
        `photo-${existing.photoId}-${variant.name}`,
      );
    }
  }
}

// Reconcile the photos[] array on the document
part.photos = (part.photos ?? []).filter((p) => submittedIds.has(p.photoId));
```

Skip new uploads (their IDs aren't in `submittedIds` yet — the client doesn't know them).

## Migration: Adding a New Variant

Adding a `card` variant to a feature that previously had only `original` and `thumbnail`:

1. Update `_PHOTO_VARIANTS` to include the new variant.
2. Existing photos won't have the new variant — running a backfill job that loads each part, fetches the `original`, re-runs `processUpload` for just the new variant, and stores the result.
3. The retrieval endpoint must fall back: if `card` doesn't exist, serve `original`. Otherwise, freshly uploaded photos (post-migration) get the variant; old photos break.

## Anti-Patterns

```typescript
// ❌ Storing the user's filename as the attachment name
this._attachmentService.store(session, docId, file.originalname, file.buffer, file.mimetype);
//   `IMG 0234 (3).HEIC` — collisions, special-character bugs, can't predict variants

// ❌ Skipping randomUUID, using a sequence number
const photoId = part.photos.length + 1;
//   Concurrent uploads collide on `1`, `2`, etc.

// ❌ Single attachment for "all variants" as JSON
this._attachmentService.store(session, docId, 'photos', JSON.stringify({original, thumbnail}), ...);
//   Loses streaming, doubles memory, no variant retrieval

// ❌ Deleting attachments on soft-delete of parent
//   Audit trail loss — soft-deleted parents still need their photos visible to admin views
```

## Quick Reference

| Operation | Method                                        | Returns                                |
| --------- | --------------------------------------------- | -------------------------------------- |
| Store     | `attachmentService.store(session, ...)`       | void; commits with `saveChanges`        |
| Read buffer | `attachmentService.get(docId, name)`        | `{data: Buffer, contentType}` or null  |
| Read stream | `attachmentService.getStream(docId, name)`  | `{stream, contentType, dispose}` or null |
| Delete    | `attachmentService.delete(session, ...)`      | void; commits with `saveChanges`        |
| Exists    | `session.advanced.attachments.exists(...)`    | boolean                                 |
