# IDS Cloud DMS — Harness Design

This is the "map of the map" — a reference for understanding why each memory layer exists, what it holds, and how it should grow or shrink over time.

---

## Workshop Memory Model

Each layer has a distinct purpose and a distinct token budget.

| Layer | Location | What it holds | When it's loaded |
|---|---|---|---|
| Working memory | Context window | Current task, code being edited, tool results | Always (current session only — ephemeral) |
| Team policy memory | `docs/standards/` | Coding rules, architecture rules, testing expectations | On demand — mandatory before editing the relevant layer |
| Procedural memory | `.claude/skills/` | How to do specific tasks correctly in this codebase | On demand — triggered by task type (see CLAUDE.md skills table) |
| Task memory | `.ai-plan/` | Implementation plans, approved decisions, progress tracking | On demand — loaded when working on a feature |
| Run-state memory | `.ai-plan/runs/` | RALPH phase state, gate results, iteration count, resume data | On demand — RALPH loop only |
| Session continuity memory | `.ai-plan/handoffs/` | Concise context for a fresh session to resume work | On demand — `/handoff` and `/resume-handoff` commands |
| Lessons learned memory | `.ai-memory/case-studies/` | Non-obvious failures, tool quirks, repairs, user corrections | On demand — check index before diagnosing repeated failures |
| Diagnostic memory | `.doctor/` | Network timeline, console errors, DOM snapshot, runtime context | On demand — IDS Doctor sidecar |

---

## Design Principles

**Always-loaded artifacts must be strictly bounded.**
`CLAUDE.md` and the session-start hook output are injected into every session. They must stay lean. No case study content, no handoff content, no full skill text belongs there — only pointers and the absolute minimum rules.

**Everything else is on-demand.**
Skills, plans, run-state, handoffs, and case studies are read when relevant. An agent reads the index first, then the individual file only if a match is found.

**Promotion is the primary exit for case studies.**
A lesson in `.ai-memory/case-studies/active/` is staging — temporary. The goal is to move it into a standard, skill, hook, or test where it benefits every future session automatically, loaded efficiently by trigger rather than by bulk retrieval.

**Memory is not the source of truth.**
Standards and skills describe what should be. Plans and run-state describe what was decided. Handoffs describe where to resume. Case studies describe what went wrong. None of these override what you observe in the code today — always verify before acting on a memory.

## Self-Improvement Loop

The harness is designed to make repeated mistakes progressively less likely:

```
gate failure / tool quirk / user correction
  ↓
capture in .ai-memory/case-studies/active/
  ↓
referenced in future session → check index → apply known fix → fewer repairs
  ↓
referenced in 3+ sessions (or immediately obvious candidate)
  ↓
promote to standard / skill / hook / test
  ↓
lesson loads automatically when triggered — no retrieval step needed
  ↓
fewer mistakes without consuming any retrieval budget
```

## Session Handoff Lifecycle

```
session ends (or RALPH reaches a safe phase boundary)
  │
  ▼
/handoff → .ai-plan/handoffs/handoff-YYYY-MM-DD_hh-mm-AM.md
  │         (300–500 words, reference paths only)
  │
  ▼
next session: /resume-handoff [latest | path]
  │
  ▼
staleness check → read referenced plan / run-state
  │
  ▼
summarize current state + next steps → ask before implementing
```

## File Size Budgets

| Artifact | Budget |
|---|---|
| CLAUDE.md | Lean — pointers and minimum always-needed rules only |
| session-start hook output | < 500 tokens |
| Individual case study | 150–300 words |
| Case study index | < 50 active rows, table format only |
| Handoff document | 300–500 words |
| Skill (SKILL.md) | No hard limit — loaded only when triggered |
| Plan | No hard limit — loaded only when working on that feature |

## RALPH Integration Points

RALPH auto-writes handoffs at two checkpoints:
- **Plan approved** (end of Phase 1): captures the approved plan path and key clarifications before implementation begins
- **All gates passed** (Step 6 Done): captures the final state for potential follow-up work

RALPH checks case studies before escalating a repeated gate failure:
- On the 3rd consecutive identical failure: read case study index, match on gate name and error type, apply known fix if found
- After a novel non-obvious repair: use `case-study-memory` skill to capture it
