# API Problem Details Standard

> **Tactical patterns for Claude live in `.claude/skills/problem-details-errors/`.** When a rule below changes, mirror the change in that SKILL.md so Claude's auto-loaded context stays in sync.

## What This Is

Problem Details is a standard JSON format for HTTP API error responses.

Instead of every endpoint returning different error payload shapes, all errors follow one consistent contract.

## Original Standards

- RFC 7807 (initial standard): https://datatracker.ietf.org/doc/html/rfc7807
- RFC 9457 (current standard, obsoletes RFC 7807): https://datatracker.ietf.org/doc/html/rfc9457

---

## What Problem It Solves

- Removes ad-hoc error payload formats across endpoints.
- Makes frontend and integrations handle errors consistently.
- Improves observability by standardizing key fields (`type`, `status`, `instance`, trace IDs).
- Reduces maintenance cost for API consumers and backend teams.

## Is It Industry Standard?

Yes. Problem Details (RFC 7807 / RFC 9457) is an IETF standard and is widely used across modern API platforms and enterprise systems.

## Why We Use It

- We want consistent backend error contracts from the start.
- We already use typed exceptions in NestJS, so mapping to Problem Details in one global filter is straightforward.
- It fits enterprise needs (traceability, predictable integrations, governance).

---

## Scope in IDS Cloud DMS

- Applies to error responses only (non-2xx).
- Success responses continue to use endpoint DTO contracts.
- Content type for errors: `application/problem+json`.

---
## Response Shape

All API errors must return:

```json
{
  "type": "urn:ids:error:validation",
  "title": "Bad Request",
  "status": 400,
  "detail": "One or more validation errors occurred.",
  "instance": "/api/locations/db",
  "requestId": "f0ef7f2a-7e07-4d97-b6f4-0f8f9ecf6a4f",
  "correlationId": "f0ef7f2a-7e07-4d97-b6f4-0f8f9ecf6a4f",
  "timestamp": "2026-02-15T18:35:10.123Z"
}
```
---

## Field Definitions

- `type`: Stable machine-readable error identifier. We use IDS URN values (see catalog below).
- `title`: Short human-readable summary of the error category.
- `status`: HTTP status code.
- `detail`: Human-readable detail for this occurrence.
- `instance`: Request path that failed.
- `requestId`: Request identifier generated by API.
- `correlationId`: Cross-service correlation identifier.
- `timestamp`: ISO-8601 UTC generation timestamp.

---

## IDS URN Error Catalog

Use these values for `type`:

- `urn:ids:error:validation` -> `400 Bad Request`
- `urn:ids:error:unauthorized` -> `401 Unauthorized`
- `urn:ids:error:forbidden` -> `403 Forbidden`
- `urn:ids:error:not-found` -> `404 Not Found`
- `urn:ids:error:conflict` -> `409 Conflict`
- `urn:ids:error:too-many-requests` -> `429 Too Many Requests`
- `urn:ids:error:internal` -> `500 Internal Server Error`
- `urn:ids:error:service-unavailable` -> `503 Service Unavailable`

## Mapping Rules

- Validation exceptions -> `urn:ids:error:validation`
- Authentication failures -> `urn:ids:error:unauthorized`
- Authorization failures -> `urn:ids:error:forbidden`
- Missing resources -> `urn:ids:error:not-found`
- Business/uniqueness conflicts -> `urn:ids:error:conflict`
- Unhandled exceptions -> `urn:ids:error:internal`

---

## Implementation Rules

- Controllers/services throw typed Nest exceptions.
- Global exception filter is responsible for formatting Problem Details.
- Controllers must not return ad-hoc custom error JSON.
- `type` values are part of public contract and must remain stable.
